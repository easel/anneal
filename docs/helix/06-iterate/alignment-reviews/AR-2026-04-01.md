## 3. Intent Summary

**Vision**: Anneal is a declarative host configuration engine — a single binary
that reads a manifest, diffs it against system state, and produces an executable
plan to converge the machine. Targets solo operators and small teams. Key
principle: the plan is an executable, auditable shell script composed of stdlib
operations.

**Requirements**: 17 P0 requirements covering validate/plan/apply workflow,
built-in providers (packages, users, files, services, storage, auth, system
utilities), custom shell providers, manifest composition, dependency DAG,
notify/trigger, iterators, secret management, run-as-user, and fail-stop
execution. 13 P1 requirements including composites, resource filtering, fact
gathering, macOS providers, line-in-file, and conditional resources.

**Feature Specs**: Nine feature specifications (FEAT-001 through FEAT-009)
decompose the P0 requirements into 36 user stories with acceptance criteria.

**Architecture**: SD-001 defines seven components (CLI, Config Loader, Engine,
Provider Interface, Stdlib, Embedded Interpreter, Secret Provider Chain). Four
ADRs record key decisions: plan-as-script, embedded interpreter (mvdan/sh), Go
static binary, stdlib-as-contract.

**Test Plan**: Four-tier strategy (unit, golden-file, provider integration,
reference deployment). No tests exist yet.

**Implementation Plan**: Nine phases from core engine through reference
deployment migration. No implementation exists yet.

## 4. Planning Stack Findings

### 4.1 Vision -> Requirements: ALIGNED

The PRD faithfully translates the vision's nine design principles into concrete
requirements. Every vision principle maps to at least one P0 requirement.
Non-goals are clearly stated and consistent with the vision's "what Anneal is
not" section.

### 4.2 Requirements -> Feature Specs: INCOMPLETE

**Coverage**: All P0 requirements are covered by feature specs. However:

- **PRD P0-13 (secret management)** is split across FEAT-001 (US-005) and
  FEAT-005 (US-036). The secret provider chain architecture is only in SD-001,
  not in a dedicated feature spec. The `resolve-secrets` command (pre-resolved
  secrets file) is specified in US-005 acceptance criteria but has no dedicated
  user story or feature spec.
- **PRD P0-14 (run-as-user)** is mentioned in FEAT-001 and FEAT-003 but has no
  dedicated user story with acceptance criteria. It appears as a requirement in
  provider specs but the mechanism itself (how `run_as` translates to stdlib
  operations, how privilege dropping works) is unspecified.
- **PRD P0-17 (built-in template variables)** is documented in SD-001 component
  detail but not reflected in any feature spec user story.
- **PRD P1-9 (fact gathering)** is listed in the PRD but has no feature spec.
  The implementation plan doesn't mention it either. The PRD states it is
  "Required for multi-platform support."
- **PRD P1-11 (macOS provider tier)** is listed in the PRD as required for
  sahara reference deployment's macOS targets but has no feature spec. The
  sahara reference deployment document mentions multi-OS support but the feature
  specs are Linux-only.
- **PRD P1-12 (line-in-file / block-in-file)** has no feature spec.
- **PRD P1-13 (conditional resources)** has no feature spec.

### 4.3 Feature Specs -> Architecture: ALIGNED with minor gaps

SD-001 covers all nine features' architectural needs. The component
decomposition is coherent and the technology choices are justified.

**Minor gap**: SD-001's Provider Interface section lists `pacman` as a built-in
provider, which is consistent with FEAT-003 US-019. However, ADR-004's stdlib
categories include `stdlib_pacman_install` which is not in SD-001's stdlib
listing. This is a cosmetic inconsistency — the stdlib will naturally grow with
providers.

### 4.4 Architecture -> Implementation Plan: ALIGNED with phasing concerns

The nine implementation phases map cleanly to the component architecture. The
phasing is value-driven (each phase produces a working system for a growing
resource set).

**Concern**: Phase 6 (Secret Management) is late in the sequence but secrets are
cross-cutting. FEAT-001 US-005 acceptance criteria include secret handling in
plan output (`(secret)` rendering). If secrets are deferred to Phase 6, Phases
1-5 must either stub secret references or accept that secret-dependent resources
can't be tested until Phase 6.

### 4.5 Test Plan -> Feature Specs: INCOMPLETE

The test plan defines a solid four-tier strategy but:

- **No test-to-user-story mapping**: The test plan's "Acceptance Criteria
  Verification" section provides a generic mapping table by criteria pattern,
  but does not enumerate which specific user stories and acceptance criteria are
  covered. There are 36 user stories with ~120 acceptance criteria — none are
  individually mapped.
- **Custom provider testing** is addressed but custom secret provider testing
  (PRD P1-6) is not.
- **Multi-OS testing** for package providers (dnf, pacman, brew) is not
  addressed in the tier 3 test container setup (Ubuntu 24.04 only).

### 4.6 Same-Layer Conflicts

**User story numbering conflict**: FEAT-002 and FEAT-003 both define US-010.
FEAT-002 US-010 is "Trigger a restart on config change" and FEAT-003 US-010 is
"Install system packages." User story IDs must be unique across features.

**Feature spec title mismatch**: FEAT-009 is titled "System Utilities" in the
feature spec but "Network & System" in the PRD requirement P0-4 and "Network
System" in the filename. The feature spec overview says "Network and system
providers" but the document `id` and title say "System Utilities."
